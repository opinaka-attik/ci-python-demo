

# 🛡️ GitHub — Créer un Branch Ruleset

## C'est quoi un Branch Ruleset ?

Un **Branch Ruleset**, c'est un ensemble de règles que tu appliques sur une ou plusieurs branches de ton dépôt GitHub. Il te permet de contrôler qui peut faire quoi : pousser du code, supprimer une branche, fusionner sans validation, etc. [github](https://github.com/opinaka-attik/ci-python-demo/settings/rules/new?target=branch)

> 💡 Imagine-le comme un **contrat d'équipe** : tout le monde doit respecter les règles que tu définis, sauf les personnes explicitement exemptées.

Pour le créer, tu vas dans **Settings → Rules → Rulesets → New branch ruleset** de ton dépôt.

***

## Étape 1 — Ruleset Name *(obligatoire)*

C'est le **nom de ta règle**. Il doit être clair et descriptif pour que n'importe qui dans ton équipe comprenne son rôle d'un coup d'œil.

**✅ Exemples de bons noms :**
- `protect-main`
- `require-pull-request`
- `no-force-push-prod`

**❌ À éviter :**
- `règle1`, `test`, `nouveau`

***

## Étape 2 — Enforcement Status

Ce paramètre contrôle si ta règle est **active ou non**. [github](https://github.com/opinaka-attik/ci-python-demo/settings/rules/new?target=branch)

| Valeur | Ce que ça fait |
|---|---|
| `Disabled` *(par défaut)* | La règle existe mais n'est pas appliquée. Utile pour la préparer avant de la lancer. |
| `Active` | La règle est appliquée immédiatement sur les branches ciblées. |

> 🎯 **Conseil** : Commence par `Disabled` pour configurer sans risque, puis passe en `Active` quand tu es prêt.

***

## Étape 3 — Bypass List

La **Bypass List** te permet d'ajouter des personnes, équipes ou applications qui **échappent aux règles**. [github](https://github.com/opinaka-attik/ci-python-demo/settings/rules/new?target=branch)

- Clique sur **"Add bypass"** pour ajouter un rôle, une équipe ou une app GitHub.
- **Cas typique** : tu veux qu'un bot CI/CD puisse pousser directement sur `main` sans passer par une Pull Request.

> ⚠️ **Attention** : Plus la liste est grande, moins ta règle est efficace. N'ajoute que ce qui est strictement nécessaire.

***

## Étape 4 — Target Branches

C'est ici que tu définis **sur quelles branches ta règle s'applique**. [github](https://github.com/opinaka-attik/ci-python-demo/settings/rules/new?target=branch)

Clique sur **"Add target"** et choisis un critère :

| Critère | Quand l'utiliser |
|---|---|
| `Default branch` | Pour protéger automatiquement ta branche principale (`main`) |
| `All branches` | Pour appliquer la règle partout |
| `Include by pattern` | Pour cibler des branches spécifiques, ex. `release/*` |

> 📝 **Exercice** : Cible uniquement la branche `main` de ton dépôt `ci-python-demo`.

***

## Étape 5 — Rules (les règles)

C'est le **cœur du formulaire**. Tu coches les règles que tu veux activer. Voici chaque option expliquée  : [github](https://github.com/opinaka-attik/ci-python-demo/settings/rules/new?target=branch)

### 🔒 Règles de base

| Règle | Ce qu'elle fait | Activée par défaut ? |
|---|---|---|
| **Restrict creations** | Seuls les utilisateurs autorisés peuvent créer des branches correspondantes | ❌ |
| **Restrict updates** | Seuls les utilisateurs autorisés peuvent pousser des mises à jour | ❌ |
| **Restrict deletions** | Seuls les utilisateurs autorisés peuvent supprimer des branches | ✅ |
| **Block force pushes** | Interdit le `git push --force` sur les branches ciblées | ✅ |

> ⚠️ Les deux règles cochées par défaut (`Restrict deletions` et `Block force pushes`) sont des **protections minimales essentielles** — garde-les activées dans presque tous les cas.

***

### 📋 Règles sur la qualité du code

| Règle | Ce qu'elle fait |
|---|---|
| **Require linear history** | Interdit les merge commits — oblige l'utilisation de `rebase` pour garder un historique propre |
| **Require signed commits** | Chaque commit doit être signé cryptographiquement (GPG/SSH) pour vérifier l'identité de l'auteur |
| **Require a pull request before merging** | Toute modification doit passer par une Pull Request avant d'être fusionnée dans la branche protégée |
| **Require status checks to pass** | Des vérifications CI/CD (tests, linting…) doivent réussir avant qu'un merge soit autorisé |

> 🚀 **Bonne pratique** : Pour un projet professionnel, active toujours **Require a pull request before merging** + **Require status checks to pass**. C'est la combinaison la plus utilisée en entreprise.

***

### 🔍 Règles avancées

| Règle | Ce qu'elle fait |
|---|---|
| **Require deployments to succeed** | Un déploiement vers un environnement défini (ex. `staging`) doit réussir avant le merge |
| **Require code scanning results** | Un outil d'analyse de sécurité du code (ex. CodeQL) doit avoir tourné et fourni ses résultats |
| **Require code quality results** | Des critères de qualité de code doivent être respectés avant de fusionner une PR |
| **Automatically request Copilot code review** | GitHub Copilot est automatiquement demandé comme reviewer sur chaque nouvelle PR |

***

## Étape 6 — Créer le Ruleset

Une fois tous les champs remplis, clique sur le bouton **"Create"** en bas du formulaire. [github](https://github.com/opinaka-attik/ci-python-demo/settings/rules/new?target=branch)

> ✅ **Checklist avant de valider :**
> - [ ] Le nom est descriptif et unique
> - [ ] Le statut est bien choisi (`Disabled` pour tester, `Active` pour déployer)
> - [ ] Les branches cibles sont configurées
> - [ ] Les règles cochées correspondent à ton besoin

***

## 🧪 Exercice pratique

Sur ton dépôt `ci-python-demo`, crée un Branch Ruleset avec les paramètres suivants :

1. **Nom** : `protect-main`
2. **Enforcement** : `Active`
3. **Target** : branche `main` (Default branch)
4. **Règles à activer** :
   - ✅ Restrict deletions
   - ✅ Block force pushes
   - ✅ Require a pull request before merging
5. Clique sur **Create** et vérifie que le ruleset apparaît bien dans la liste des règles.
